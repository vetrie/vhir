VHIR Technical Reference

A FHIR-inspired, standalone interoperability standard for veterinary medicine — JSON + REST, built for clinics, farms, and the systems connecting them.

REST / JSON 20 Resource Types Herd-First Pet Insurance Microchip Lookup

What is VHIR?

Human medicine has FHIR (Fast Healthcare Interoperability Resources) — a widely adopted standard for exchanging electronic health records. Veterinary medicine has no equivalent. The ~100 Practice Information Management Systems (PIMS) used by vet clinics worldwide all speak different data formats, making cross-system workflows (referrals, pharmacies, insurance, lost-pet registries) brittle or impossible.

VHIR is a standalone JSON + REST standard that solves this. It borrows FHIR's proven architectural ideas — resource-oriented REST, JSON payloads, OAuth2 scopes, search parameters, Bundle envelopes — but strips away human-medicine regulatory baggage (HIPAA Provenance resources, ICD-10/CPT/RxNorm coupling, C-CDA) and adds vet-specific first-class features:

  • Herd / group treatment — administer drugs to 50 cattle in one record, with per-animal completion tracking
  • Withdrawal periods — milk, meat, and eggs hold times required by FDA for food-animal drugs
  • Multi-owner support — primary owner, co-owner, breeder, trainer, farm staff — all with different access scopes
  • Microchip federated lookup — scan a chip ID, get the registry pointer without exposing owner PII
  • Pet insurance — pre-authorization and adjudication shapes for Trupanion VetDirectPay and post-service insurers
ℹ️
Not a FHIR Implementation Guide. VHIR is its own standard. It doesn't use FHIR namespaces, resource types, or tooling. If you know FHIR, you'll find VHIR familiar but simpler. If you don't know FHIR, VHIR is designed to be learned on its own.

Veterinary Medicine Primer

This section explains vet-specific concepts that appear throughout the API. Skip it if you work with veterinary data daily.

Practice Types

🐾
Companion animal (small animal) practice treats dogs, cats, rabbits, birds, and exotic pets. Most urban clinics fall here. Visits are individual animal → owner interactions, similar to a human GP.

Large animal / equine practice treats horses, cattle, pigs, sheep, goats. Vets often travel to farms (ambulatory practice). Horses are treated individually; cattle and pigs are often treated as herds.

Mixed practice treats both. Common in rural areas.

PIMS — Practice Information Management Systems

A PIMS is the all-in-one software a clinic uses: medical records, invoicing, scheduling, inventory, reminders. Think of it as the EHR (Electronic Health Record) + practice management software combined. Examples include ezyVet, Vetspire, Cornerstone (IDEXX), Avimark, Provet Cloud. Each has its own proprietary data format — VHIR is the common language between them.

Microchips

Pet microchips are passive RFID transponders — they have no battery and don't transmit continuously. A scanner held close to the animal induces a current in the chip, which then broadcasts its ID number for one brief moment. The dominant standard is ISO 11784/11785 (FDX-B), which defines a 15-digit decimal ID. Older US chips used 125 kHz frequencies (Avid, HomeAgain pre-ISO) and are not compatible with ISO scanners.

🔬
ISO 11784 chip IDs are 15 decimal digits. The first three digits are the country or manufacturer code — 956 is the ISO country code for an international/manufacturer allocation. A chip starting with 840 is USDA-registered (US livestock).

Withdrawal Periods

When a food-producing animal (dairy cow, beef cattle, laying hen) is given a drug, there is a mandatory waiting period before its milk, meat, or eggs can enter the food supply. This is the withdrawal period. Veterinarians are legally required to record it. VHIR stores this as withdrawal: { milkHours, meatHours, eggsHours } on a MedicationRequest and withdrawalEnds: { milk, meat, eggs } (computed dates) on a MedicationAdministration.

VFD — Veterinary Feed Directive

Medically important antibiotics (e.g., tetracyclines) in animal feed require a written order from a licensed vet — a Veterinary Feed Directive. In VHIR, MedicationRequest.vfd: true flags an order that must be treated as a VFD rather than a standard prescription.

Controlled Substances (DEA Schedules)

Drugs like ketamine (common equine anesthetic), opioids, and barbiturates (euthanasia solutions) are DEA-scheduled. Veterinarians must maintain a DEA number and special records for Schedule II drugs. VHIR carries MedicationRequest.controlled.schedule and deaOrderFormNumber for this purpose.

Quick Start

Run the Reference Server

The server requires Postgres 16. The included docker-compose.yml handles everything:

shell bash
# Clone the repo and start Postgres + server
git clone https://github.com/vhir/vhir
cd vhir/server
docker compose up -d postgres
uv sync
uv run uvicorn vhir_server.main:app --reload --port 8000

The server starts on http://localhost:8000. Interactive API docs are at /v1/_docs.

💡
Dev token mode is on by default. Requests without an Authorization header automatically receive system/*.read system/*.write permissions. You can curl the API immediately without setting up OAuth2. Disable with DEV_TOKEN_MODE=false.

Your First API Calls

Create an animal bash
curl -s -X POST http://localhost:8000/v1/Animal \
  -H "Content-Type: application/json" \
  -d '{
    "resourceType": "Animal",
    "name": "Biscuit",
    "species": "canis-familiaris",
    "breed": "Labrador Retriever",
    "sex": "male",
    "neuterStatus": "neutered",
    "identifiers": [{"type": "microchip-iso", "value": "956000004287442"}]
  }' | jq '.'
Response json
{
  "resourceType": "Animal",
  "id": "01HX7BKZM0ABCDEF123456789",   // ULID — server-assigned
  "meta": {
    "version": 1,
    "lastUpdated": "2026-05-17T14:22:10Z"
  },
  "name": "Biscuit",
  "species": "canis-familiaris",
  "identifiers": [{"type": "microchip-iso", "value": "956000004287442"}]
}
Search by microchip bash
curl -s "http://localhost:8000/v1/Animal?identifier=microchip-iso|956000004287442" | jq '.total'
Look up a lost pet chip bash
curl -s -X POST http://localhost:8000/v1/\$lookup-microchip \
  -H "Content-Type: application/json" \
  -d '{"chipId": "956000004287442"}'

Server Stack

FastAPI

Async Python web framework. Handles routing, request validation (via Pydantic), and auto-generates OpenAPI docs at /v1/_docs.

Pydantic v2

Request/response models with runtime validation. All 20 resource types are defined in core/models.py.

SQLAlchemy 2.0 async + asyncpg

Async Postgres driver. All DB operations are non-blocking. ResourceRepository wraps all CRUD operations.

Postgres 16 JSONB

Each resource type has its own table with a JSONB body column. GIN indexes on frequently-searched array fields (identifiers, owners, members).

Storage Layout

One table per resource type keeps indexes natural and avoids type-dispatch overhead. All tables share the same five columns:

SQL sql
CREATE TABLE animal (
  id           TEXT PRIMARY KEY,             -- ULID, server-assigned
  version      INTEGER NOT NULL,             -- monotonically increasing
  last_updated TIMESTAMPTZ NOT NULL,
  body         JSONB NOT NULL,               -- full resource JSON
  deleted      BOOLEAN NOT NULL DEFAULT false -- soft delete
);

-- GIN indexes for array fields frequently searched via JSONB containment
CREATE INDEX ix_animal_identifiers ON animal USING GIN ((body->'identifiers'));
CREATE INDEX ix_animal_owners      ON animal USING GIN ((body->'owners'));

A separate resource_history append-only table records every create / update / delete with the full body snapshot, enabling audit trails.

Resource IDs

All resource IDs are ULIDs (Universally Unique Lexicographically Sortable Identifiers). ULIDs are 26-character Base32 strings that embed a millisecond timestamp prefix, making them naturally sortable by creation time — useful for pagination. The server always assigns IDs; clients never specify them.

Example ULID
01HX7BKZM0ABCDEFGHJ12345AB
└── timestamp ──┘└── random ─┘

REST Conventions

Endpoints

Every resource type exposes a standard five-operation surface:

POST/v1/{Type}Create (server assigns ULID, returns 201)
GET/v1/{Type}/{id}Read by ID (returns 200 or 404)
PUT/v1/{Type}/{id}Full replacement update (returns 200)
DELETE/v1/{Type}/{id}Soft delete — sets deleted=true (returns 204)
GET/v1/{Type}?param=valueSearch → Bundle searchset (returns 200)

Content Type

VHIR uses the MIME type application/vhir+json. The reference server also accepts and returns application/json for compatibility. Always send Content-Type: application/json (or application/vhir+json) on POST / PUT requests.

Optimistic Concurrency

Every response includes an ETag header set to the current version number in quotes (e.g., ETag: "3"). To prevent accidental overwrites, send If-Match: "3" on a PUT request. If the version has changed since you read the resource, the server returns 412 Precondition Failed.

Concurrent update example bash
# Read returns ETag: "2"
curl -s http://localhost:8000/v1/Animal/01HX...

# Update — must include If-Match with the version you read
curl -s -X PUT http://localhost:8000/v1/Animal/01HX... \
  -H "If-Match: \"2\"" \
  -H "Content-Type: application/json" \
  -d '{...}'

# If someone else updated it first, you get 412 Precondition Failed

References

References between resources use the format "ResourceType/id" in a ref field:

json
{
  "subject": { "ref": "Animal/01HX7BKZM0ABCDEF123456789" },
  "requester": { "ref": "Practitioner/01HX7BKZ...", "display": "Dr. Smith DVM" }
}

Soft Deletes

DELETE sets deleted = true in the database; the row is never physically removed. A deleted resource returns 404 on GET. The history table retains all versions including the delete event. To restore a deleted resource, you must create a new one — there is no undelete endpoint in v0.

Resource Envelope

Every VHIR resource shares a common envelope structure. Fields in meta are server-managed and will be overwritten if you include them in a POST / PUT body.

json
{
  "resourceType": "Animal",        // Always the resource type name
  "id": "01HX7BKZM0ABCDEF123456789", // ULID — server-assigned
  "meta": {
    "version": 3,                     // Increments on every update
    "lastUpdated": "2026-05-17T14:32:11Z",
    "profiles": ["vhir/Animal-Canine"] // Optional conformance profiles
  },
  "extensions": {               // Vendor-specific data — open-content slot
    "com.ezyvet/internalId": "12345"
  },
  // ... resource-specific fields
}

Bundle — Search Response Envelope

Search operations return a Bundle with type: "searchset":

json
{
  "resourceType": "Bundle",
  "type": "searchset",
  "total": 47,              // Total matching records (not just this page)
  "entry": [
    { "resource": { /* Animal */ } },
    { "resource": { /* Animal */ } }
  ]
}

Authentication

Dev Token Mode (Default)

When DEV_TOKEN_MODE=true (the default), requests without an Authorization header automatically receive full system-level permissions. This lets you explore the API with plain curl without configuring OAuth2.

To get a dev JWT token explicitly (for testing auth code paths):

bash
curl -s -X POST http://localhost:8000/oauth/token \
  -H "Content-Type: application/json" \
  -d '{"grant_type":"dev","subject":"vet-user","role":"veterinarian","org_id":"clinic-1"}'

VHIR-SMART Scopes

The scope format mirrors SMART-on-FHIR: {context}/{Resource}.{action}

ScopeMeaning
system/*.readRead any resource (server-to-server, admin)
system/*.writeWrite any resource
user/Animal.readRead animals accessible to the logged-in user
animal/Observation.readRead observations for a specific animal context
owner/MedicationRequest.readRead prescriptions for animals owned by the token owner
⚠️
Set DEV_TOKEN_MODE=false and configure a real JWT secret before exposing the server outside localhost. See vhir_server/config.py for all environment variables.

Identifier Search Syntax

The identifier search param searches inside the identifiers[] array. Use pipe notation to match both type and value simultaneously:

bash
# Match by type AND value (preferred — most precise)
curl "http://localhost:8000/v1/Animal?identifier=microchip-iso|956000004287442"

# Match by value only (searches across all identifier types)
curl "http://localhost:8000/v1/Animal?identifier=956000004287442"

# USDA 840 livestock EID
curl "http://localhost:8000/v1/Animal?identifier=usda-840|840000012345678"

M0Observation

Any measured or observed clinical finding: vital signs, lab results, body condition score (BCS), imaging findings, or wearable device data.

Panel Observations (component[])

A vital signs panel records multiple measurements as a single Observation using component[]. This is preferable to creating five separate Observations for a routine vitals check:

Vital signs panel json
{
  "resourceType": "Observation",
  "status": "final",
  "category": ["vital-signs"],
  "code": { "code": "vitals-panel", "display": "Vital signs panel" },
  "subject": { "ref": "Animal/01HX..." },
  "effectiveDateTime": "2026-05-17T14:10:00Z",
  "component": [
    { "code": { "code": "body-weight" }, "valueQuantity": { "value": 32.5, "unit": "kg" } },
    { "code": { "code": "temperature" },  "valueQuantity": { "value": 38.6, "unit": "C"  } },
    { "code": { "code": "heart-rate"  },  "valueQuantity": { "value": 88,   "unit": "/min" } }
  ],
  "interpretation": "normal"
}
🌡️
Normal vital sign ranges differ by species. A healthy dog's temperature is 38.3–39.2 °C; a horse's is 37.2–38.3 °C; a rabbit's is 38.5–40 °C. interpretation should always be set by the recording clinician against the appropriate species baseline, not derived automatically.

M1Group

A herd, flock, litter, or colony — a named collection of animals treated as a unit. Group is VHIR's biggest differentiator from FHIR. MedicationAdministration can target a Group directly, with per-member completion tracked in the same record.

🐄
In food animal medicine, a single veterinarian may treat hundreds of cattle in an afternoon. Recording one MedicationAdministration per animal would create thousands of records. The Group model lets you record the treatment at the herd level and track per-animal completion (and refusals) within a single resource — matching how vets actually work and what FDA record-keeping requires.

Example

POST /v1/Group — dairy herd json
{
  "resourceType": "Group",
  "name": "Pen 4",
  "type": "animal",
  "productionPurpose": "dairy",
  "premisesId": "US-PIN-WI-88888",
  "managingOrganization": { "ref": "Organization/01HX..." },
  "members": [
    { "ref": "Animal/01HX..." },
    { "ref": "Animal/01HY..." }
  ]
}

Production Purposes

productionPurpose values: dairy, beef, feedlot, layer, broiler, breeding, companion-litter, other

M1Procedure

M1MedicationDispense

Pharmacy fulfillment of a MedicationRequest — records that the drug was physically dispensed (sent home with the owner or transferred to the farm). Always links back to the originating MedicationRequest via authorizingPrescription.

M1MedicationAdministration

Records that a drug was actually given to an animal (or herd). This is distinct from MedicationRequest (the order) and MedicationDispense (dispensing). The key vet-specific feature is herd administration: when subject is a Group, memberCompletion[] tracks the status per animal.

Herd Administration

POST /v1/MedicationAdministration — group treatment with withdrawal json
{
  "resourceType": "MedicationAdministration",
  "status": "completed",
  "medication": { "name": "Penicillin G 300,000 IU/mL", "form": "injectable" },
  "subject": { "ref": "Group/01HX..." },  // ← targets the whole herd
  "performer": [{ "ref": "Practitioner/01HX..." }],
  "effectiveDateTime": "2026-05-17T07:00:00Z",
  "withdrawalEnds": {          // Computed end dates for regulatory records
    "meat": "2026-06-01",
    "milk": "2026-05-18"
  },
  "memberCompletion": [        // Per-animal tracking within the group record
    { "animal": "Animal/cow-101", "status": "completed", "effectiveDateTime": "2026-05-17T07:01:00Z" },
    { "animal": "Animal/cow-102", "status": "completed", "effectiveDateTime": "2026-05-17T07:04:00Z" },
    { "animal": "Animal/cow-103", "status": "refused" }   // This one kicked
  ]
}
📋
FDA regulations (21 CFR 530) require food animal withdrawal records to be kept for 2 years. withdrawalEnds stores the computed calendar dates — not just the number of hours — so they're immediately queryable. A farm management system can run GET /v1/MedicationAdministration?subject=Group/herd-1 and compute which animals are still in withdrawal today.

M1Appointment

A booked visit. References Animal as subject and Practitioners as participants. Can reference a Slot from the scheduling system.

M1Schedule & Slot

Schedule represents a practitioner's or room's calendar for a planning horizon. Slot is a time block within that Schedule. Appointments consume Slots. This three-resource model matches how clinic scheduling actually works: first you define when Dr. Kim is available (Schedule), the system creates free Slot blocks, and Appointment bookings mark those Slots as busy.

M2InsuranceClaim

Pet insurance claim supporting the two dominant US workflows: Trupanion VetDirectPay (pre-authorization at time of service, insurer pays clinic directly) and post-service reimbursement (owner pays clinic, then claims from insurer — Nationwide, Pets Best, Hartville, ASPCA).

💳
VetDirectPay (pre-auth): Owner presents at clinic → clinic submits pre-auth request with estimated cost → Trupanion responds with an approved amount → clinic performs treatment → Trupanion pays clinic directly. The owner only pays the deductible/co-pay.

Post-service reimbursement: Owner pays the full invoice → submits a claim to their insurer with the invoice → insurer reimburses owner. More common among US pet insurers.

$lookup-microchip Operation

The system operation POST /v1/$lookup-microchip implements a two-step federated microchip lookup:

  1. Local search — searches the VHIR server's own Animal table for a matching identifier. If found, returns the Animal/ reference.
  2. Broker fallback — for ISO 15-digit chips not found locally, returns the AAHA petmicrochiplookup.org broker URL so the caller can perform the registry lookup. Owner PII is never returned — only the registry pointer.
ℹ️
This is a system operation (like FHIR's $everything), not a resource CRUD endpoint. It starts with $ in the URL. Escape the dollar sign in bash with \$ or single quotes.
Found locally json
// POST /v1/$lookup-microchip
{ "chipId": "956000004287442" }

// Response
{
  "found": true,
  "localAnimal": "Animal/01HX7BKZM0ABCDEF123456789"
}
Not found locally — broker response json
{
  "found": false,
  "broker": "https://petmicrochiplookup.org",
  "registryContact": {
    "system": "url",
    "value": "https://petmicrochiplookup.org"
  }
}

Walkthrough: Companion Animal GP Visit

A complete workflow from registration to diagnosis and prescription — the typical small animal clinic visit.

walkthroughs/companion-visit.sh bash
# 1. Create owner
OWNER=$(curl -s -X POST $BASE/Owner -d '{"resourceType":"Owner","name":{"given":"John","family":"Smith"},...}')
OWNER_ID=$(echo $OWNER | jq -r '.id')

# 2. Create animal linked to owner
ANIMAL=$(curl -s -X POST $BASE/Animal -d "{...\"owners\":[{\"ref\":\"Owner/$OWNER_ID\",\"role\":\"primary\"}]}")
ANIMAL_ID=$(echo $ANIMAL | jq -r '.id')

# 3. Create encounter
ENC=$(curl -s -X POST $BASE/Encounter -d "{...\"subject\":{\"ref\":\"Animal/$ANIMAL_ID\"}}")

# 4. Record vitals as Observation
# 5. Post diagnosis as Condition
# 6. Write prescription as MedicationRequest
# 7. Search all records for the animal
curl "$BASE/Condition?subject=Animal/$ANIMAL_ID"
curl "$BASE/MedicationRequest?subject=Animal/$ANIMAL_ID"
curl "$BASE/Observation?encounter=Encounter/$ENC_ID"

Run the full script: bash docs/walkthroughs/companion-visit.sh
